A philosophical pessimism of code

Posted on Jan 31, 2024 2 mins read

There are plenty of arguments to be made for philosophical optimism in code. Code is art. Code is fun. Code helps us do things.

But there’s also value in pessimism, and on the balance, experienced developers tend to have more of it.

What does philosophical pessimism look like for programmers? Let’s break it down.

Principle 1: Most code blows up; therefore, limit the blast radius.

Programmers know it’s rare for code to work perfectly on the first (or second or third) try, yet all too often we write it as though it will be operating in a predictable, frictionless environment.

Instead, it’s better to assume every piece of code you write is a ticking time bomb. Put it in the smallest, tightest box you can, test it thoroughly, and try to keep it away from everything else. Make sure that when it blows up, you’ll only have to fix it in one place, not everywhere.

In practice, this is all about information hiding, encapsulation, and the Single-Responsibility Principle.

Principle 2: Most code gets thrown away; therefore, don’t write it without supporting evidence.

  • Most code blows up, so limit the blast radius
  • Most code gets thrown away, so don’t write it without evidence
  • Most code is useless, so at least make it usable
  • Most code is wrong, so be wrong as quickly and as
