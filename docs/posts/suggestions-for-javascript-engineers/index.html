Some suggestions for JavaScript engineers

Regarding manners, etiquette, and lively conversation

Posted on Aug 26, 2021 9 mins read

For years there’s been a bitter debate in the development community regarding JavaScript’s D&D alignment. Although few are brazen enough to argue that it’s Lawful, a full spectrum of opinions exists about whether it’s Good or Evil. In a surprising reversal, I’m told Jeff Atwood has recently taken the side of Evil, although he’s probably just trying to protect his “bad boy” reputation. I won’t share my opinion on the subject but perhaps a bit of history can shed some light.

JavaScript was invented by Billy Eichner, a C-list actor and comedian (although it’s never been clear if JavaScript was intended as a stage production or a joke). It was a clever solution to an essential problem: web browsers at the time weren’t able to execute arbitrary malware exploits on users’ computers, which limited the general appeal of the Internet. JavaScript wasn’t expected to become the world’s most popular programming language; back in 2005, the Internet was still a fledgling technology, dwarfed by the popularity and usefulness of the average Coke machine. These days, Coke machines are just about the only things that don’t have a WiFi chip, and JavaScript is so fashionable that even the average middle-school student knows what it is (although technical recruiters are still very much in the dark). Eichner named JavaScript after Java, hoping to make it seem less threatening to traditional Java engineers, but since the Java trademark has since been acquired by the military-industrial complex, JavaScript has been renamed to EczemaScript. This designation comes in honor of the intense dry skin and rashes reported by its earliest adopters.

“I’ll have the JavaScript, please.”

~ A technical recruiter at Red Lobster

Shortly after JavaScript was released, jQuery came in its wake, rapidly receiving FDA approval for treatment of the conditions caused by the former. Now engineers could trade eczema for bloating and vertigo, which made their discomfort considerably less visible on first approach. It was a good trade, but led to an undeserved feeling of completion among those who maintained the language. Douglas Crockford, creator of the JSON format, said at 2008’s W3C Membership Meeting that JavaScript was “both perfect and complete, neither lacking nor having in excess any essential feature.” His 800-plus-page book, JavaScript: All of it is Good, was released later that year. It was received as one of the decade’s most prodigious technical manuals despite its tendency to describe the language in inappropriately sexual terms.

Inside of a few years, it became clear that all the high-fiving and long vacations had been premature. Although JavaScript was peerless for its ability to distribute malware, it fell short when it came to creating SPAs (Supercomplex Purchasing Apps), which were the wave of the future. After all, why would you ask a user to download and install an application to watch ads on their computer when you could simply make them watch ads on the web? But without a consistent methodology or static type system, complex JavaScript apps quickly went the way of the dojo, full of infighting and failed attempts to create standards.

Almost every engineer in the world had the same idea at the same time: to create a new language or framework or toolkit that would standardize the development of web apps and make them easier to build, putting a permanent end to the problem of complexity on the web. And each of those engineers built and released their solution over the course of the next five years. Rather than choose between them, modern web engineers use every single one in every single project—to see how this works, visit your nearest node_modules folder.

Needless to say, things are a lot simpler now.

The days when JavaScript stood on its own are long gone. And the culture that has built up around the New Web has its own system of taboos and mannerisms which are essential for the modern engineer to understand.

Below is your starter kit for navigating web development culture in 2020.

1. Pretend like it’s no big deal

If someone asks about your tech stack, the correct response is, “oh, it’s pretty basic. We’re using Node on the back end and React JSX RxJS Redux Bootstrap Lodash Babel Webpack Jest Yarn Yarn Yarn ERROR REBOOT on the front end, with Git for version control.” If they don’t know what one of those things is, cock your head and frown. “I thought everybody had used Ramda once or twice.” If you’re interviewing them for a job, now’s a good time to abruptly thank them for coming in and hold the door open for them to leave.

Junior engineers, in particular, should be expected to show up on their first day with a spectacular knowledge of ten or more major web technologies. I mean, back when you started, you only had to know two or three. But that was back when there was still ice at the North Pole. So, you know. Ancient history.

2. Don’t write code

As recently as 2011 it was appropriate to Google things like “how to iterate through object values” or “how to add padding to the left side of a string.” We still Google those things in modern web development, but we prefix each query with “npm” and install the first package that comes up, which short-circuits the entire process of copying and pasting the relevant lines of code from Stack Overflow. It’s a productivity gain of several seconds per week.

You might worry that borrowing all your code from strangers will gradually erode your web development skills. But worrying doesn’t solve problems, whereas borrowing code ostensibly does.

3. Choose a hill to die on

You’ve heard of two-party politics? Those parties are Angular and React. Pick one, vote for it in every user survey, and defend it until your dying day regardless of the technical decisions it may make. Use dubious statistics to prove that the other one is on the decline (probably because it performs so poorly in Real-World Scenarios™). Request your tombstone to be engraved with the words “Users can tell what framework a web app was developed with.” Use job boards specific to your framework preference. This is the way of the web developer.

If you ever meet someone who prefers a third framework, make sure to explain to them that it doesn’t have a big enough ecosystem. They probably don’t know.

4. Complain a lot about web development these days

The only way to make friends on Hacker News is to talk about how unnecessary all the packages and frameworks and tools we use in modern web development are. You can select from the following useful points as you make your case:

  • You used to develop in plain HTML, JavaScript and CSS and it worked just fine. Your websites were compatible with every browser, they loaded in milliseconds, and the architecture never got out of hand.
  • Your company decided to roll their own JavaScript framework, and anyone who doesn’t know how to do that isn’t a real developer anyway.
  • You found this one website that is TEN MEGABYTES LARGE. Seriously, Netflix, what are you even doing over there? LOL ridiculous.
  • Using a framework to develop a website is like using a landmine to swat a housefly. Literally. All websites are the same size and none of them need a framework.
  • Native apps are where all the real development is happening, anyway.

You’ll get so many upvotes you won’t know what to do with them. Probably cash them in and buy some Dogecoin. (I know, I know. Upvotes are actually worthless. But so is Dogecoin, therefore…?)

5. Talk a lot about unit tests, but never write any

Unit-testing a <button> element to make sure it calls a method when you click it is three things:

  1. a great way of making it more difficult to change the name of the method your button calls;
  2. the only thing your framework’s unit-testing documentation will tell you how to do;
  3. not so much a unit test for your own code as for your preferred web browser’s fork of Chromium. (I guess we better test that thing, since we’re stuck with it.)

It’s important to act like you spend six hours a day writing unit tests. This is the way of the web developer. But the moment you exit your Hacker News tab, you’ll realize the site you’re building is little more than a fancied-up table where each cell is a button that submits an AJAX request. (Ooh, and don’t forget that cell over there—it has a CSS transition.) Anything you might want to test is already part of the W3C specification. If your tests actually found a bug, and if it didn’t turn out to be a typo, the most you could do about it is put in for a bug bounty from Google.

So yeah, you won’t be writing any unit tests today. But not because you don’t believe in unit tests. Unit tests are part of good code. All code should have unit tests. 100% code coverage! You should get an iron-on patch or something.

6. Find someone to gatekeep who isn’t you

Back-end devs, embedded systems devs, native app devs, and whatever kind of devs use “Kubernetes,” all spend so much time telling web devs they’re not real devs that sometimes it can feel like you’re missing out on a party—one where all the attendees are massive jerks who like to make up their own definitions of “programming” because they weren’t loved enough as children.

It doesn’t have to be that way. There are other devs you can be condescending toward: HTML devs, CSS devs, and people who use no-code platforms. And if they can find their own hapless victims to be condescending toward, then we can all be jerks together. Being awful with a group is just more fun.

Someone once tried to tell me that anyone who creates software is a developer. I scoffed right away. Where would that leave us? Where would I get my superiority dopamine? You might as well tell me that anyone who plays the guitar is a musician. Ha! I can play the solo from Stairway to Heaven (the starting one, not the middle one). If that doesn’t make me better than someone, why even do it?

Hopefully these points of etiquette will help you fit in at your new web development job. If not, you’re probably doing them wrong. Good luck!
